The format + security — the meta-thinking

4 min
DOSection 07 (the two-column “why a page / how we’d secure it”).
  • I made a deliberate choice to build this as a web page, not a deck. Here’s why it matters in the AI age: a page like this can be generated and personalized per customer in minutes; it updates at a link instead of being a fixed file you email around; and the reader picks their own view, so a CFO sees the CFO’s version and a CTO sees theirs, instead of everyone getting the same slides.
  • But I’d rather name the hard parts than pretend they’re solved — so the other column is how you’d run this securely at Stripe: not a public link for a real customer, nothing confidential baked into the page, a real security review. A deck is ‘safe’ because it’s inert; this is more powerful but asks more of us. That’s a tradeoff worth making on purpose.

Q

Q&A — likely questions + crisp answers

  • “How did you actually build this / how is it hosted?” One-liner: It’s a single HTML file in a GitHub repository. GitHub has a free feature called Pages that serves whatever’s in the repo as a live website. So the file is the site — no separate build. I update the file, the URL updates itself. If they want more: The repo is like a folder in the cloud. GitHub Pages publishes it. index.html is special — it’s the default page a server shows at the base URL, like a store address taking you to the front door. The other pages — the briefing, the pre-read — are just more files in the same folder, each with its own URL (…/the-briefing.html), and they link to each other by filename. Key point (clears the common confusion): the pages aren’t generated from index.html — they’re siblings, separate files sitting next to each other. index.html is just the one shown by default. If technical (“what’s the build process?”): There isn’t one — plain HTML, styling and fonts embedded in the file, no framework, no compile step. GitHub Pages serves the static files as-is. That’s why it’s portable and low-risk. Your live setup (for reference): repo danielle-chauncey/thd-stripe-engagement, branch main, root folder, 4 HTML files + README, URL https://danielle-chauncey.github.io/thd-stripe-engagement/.
  • “Could Stripe really use this with a real customer?” Not as a public link — behind SSO or an expiring link, nothing confidential in the page itself, security-reviewed. Section 7 walks through it.
  • “Isn’t a fictional plan risky without real account data?” Everything’s public-sourced and I flag every assumption. The point isn’t the specific numbers — it’s the repeatable method.
  • “What would you do differently with real Stripe data?” Gather the four things only the account team and Home Depot can tell me — their real stack, the contractor revenue, their definition of friction — and the content auto-updates from there.
